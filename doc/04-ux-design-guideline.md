## 1. 디자인 원칙

### 1.1 핵심 가치

DualTetraX 앱은 **뷰티 디바이스의 동반자**이다. 디바이스를 사용하는 경험의 연장선에서 사용자가 자신의 뷰티 루틴을 관리하고, 성취감을 느끼며, 더 나은 습관을 형성하도록 돕는 것이 앱의 존재 이유이다.

| 원칙 | 설명 | 구현 방향 |
|------|------|----------|
| **단순함 우선** | 40-50대 여성이 별도 안내 없이 사용할 수 있어야 한다 | 한 화면에 하나의 핵심 행동, 3탭 이내 도달 |
| **아름다움과 신뢰** | 프리미엄 뷰티 디바이스에 걸맞은 세련된 비주얼 | 절제된 색상, 넉넉한 여백, 부드러운 전환 |
| **격려와 성취** | 사용 기록을 통해 꾸준한 루틴을 응원한다 | 긍정적 언어, 달성률 시각화, 칭찬 메시지 |
| **투명한 연결** | 디바이스-앱-클라우드 연결 상태를 항상 명확히 보여준다 | 연결 아이콘 상태, 동기화 진행률, 오류 안내 |

### 1.2 Mobile-First 접근

- **모든 기능은 모바일 앱에서 완결**된다. 웹 대시보드는 보조 조회/관리 수단일 뿐이다.
- 모바일 앱(Flutter)이 유일한 데이터 수집 경로이다 (BLE 연결 -> 세션 수집 -> 서버 동기화).
- 웹 대시보드는 PC에서 넓은 화면으로 통계를 확인하거나, 관리자가 전체 현황을 볼 때만 사용한다.
- 디자인 작업 순서: 모바일 화면 설계 완료 후, 웹 대시보드를 모바일 디자인의 확장으로 구현한다.

### 1.3 접근성 기준

뷰티 디바이스의 주 사용자층인 20-50대 여성을 모두 포괄하기 위한 최소 접근성 기준이다.

| 항목 | 기준 | 근거 |
|------|------|------|
| **터치 타겟 크기** | 최소 48x48dp (권장 56x56dp) | Material Design 3 / Apple HIG |
| **터치 타겟 간격** | 최소 8dp | 오조작 방지 |
| **텍스트 최소 크기** | 본문 14sp 이상, 캡션 12sp 이상 | 40대 이상 시인성 확보 |
| **명암비 (텍스트)** | WCAG AA 기준: 일반 4.5:1, 큰 텍스트 3:1 | 시각 접근성 |
| **명암비 (UI 요소)** | 3:1 이상 | 버튼, 아이콘, 차트 구분선 등 |
| **색맹 대응** | 색상만으로 정보를 전달하지 않음 | 아이콘/패턴/텍스트 레이블 병행 |
| **동적 글꼴** | 시스템 글꼴 크기 설정 존중 | iOS Dynamic Type, Android sp 단위 |

### 1.4 인지 부하 최소화

사용자의 의사결정 횟수를 줄이고, 한 번에 처리해야 하는 정보량을 통제한다.

- **점진적 공개(Progressive Disclosure)**: 상세 정보는 한 단계 깊이에 숨기고, 요약을 먼저 보여준다.
- **일관된 패턴**: 모든 화면에서 같은 위치에 같은 유형의 정보를 배치한다.
- **기본값 제공**: 설정값에는 합리적인 기본값을 제공하여 사용자가 별도 설정 없이도 바로 사용할 수 있게 한다.
- **Miller의 법칙**: 한 화면에 동시에 표시하는 독립적 정보 그룹은 5개 이내로 제한한다.
- **피드백 즉시 제공**: 버튼 탭 시 시각적 반응(리플 효과, 색상 변화)은 100ms 이내에 나타나야 한다.

### 1.5 대상 사용자 페르소나

#### 페르소나 A: 김지은 (28세, 마케터)
- 스마트폰에 익숙하고 SNS 활동이 활발하다
- 트렌디한 디자인과 빠른 반응을 기대한다
- 세부 통계보다 간결한 요약과 공유 가능한 형태를 선호한다
- "오늘도 루틴 완료!" 같은 성취감 있는 경험을 원한다

#### 페르소나 B: 이미영 (48세, 주부)
- 카카오톡과 네이버 정도만 사용한다
- 영어 메뉴나 복잡한 설정에 불안감을 느낀다
- 큰 글씨와 명확한 안내 문구를 선호한다
- "이 버튼을 누르면 무슨 일이 일어나는지" 확신이 있어야 행동한다

**설계 원칙**: 페르소나 B가 막힘 없이 사용할 수 있는 단순함을 기본으로 하되, 페르소나 A가 지루하지 않은 세련된 비주얼을 입힌다.

## 2. 색상 팔레트

### 2.1 브랜드 색상

DualTetraX는 프리미엄 뷰티 디바이스이므로, 색상은 **깨끗함**, **신뢰감**, **세련됨**을 전달해야 한다. 원색(Primary)보다 약간 차분하게 톤다운된 색조를 사용한다.

```
브랜드 프라이머리:  #6366F1  (Indigo 500 계열 - 신뢰, 프리미엄)
브랜드 세컨더리:    #EC4899  (Pink 500 계열 - 뷰티, 여성적)
브랜드 악센트:      #F59E0B  (Amber 500 계열 - 에너지, 따뜻함)
```

### 2.2 기능별 색상 (Shot Type 구분)

디바이스의 3가지 핵심 기능을 일관되게 색상으로 구분한다. 웹 대시보드(`WeeklyChart.tsx`, `ShotTypePie.tsx`)에서 이미 사용 중인 색상과의 일관성을 유지하되, 모바일에서는 약간 더 생생한 톤을 적용한다.

| 기능 | 모바일 색상 | 웹 색상 (기존) | 의미 | 용도 |
|------|------------|---------------|------|------|
| **U-Shot (초음파)** | `#4F86F7` | `#3B82F6` (blue-500) | 시원한 파란색 = 초음파 진동 | 차트, 모드 아이콘, 배지 |
| **E-Shot (전기자극)** | `#F5A623` | `#F59E0B` (amber-500) | 따뜻한 주황색 = 전기 에너지 | 차트, 모드 아이콘, 배지 |
| **LED Care** | `#34D399` | `#10B981` (emerald-500) | 초록색 = 치유, 광 케어 | 차트, 모드 아이콘, 배지 |

### 2.6 그래디언트 (프리미엄 강조 요소에 제한적 사용)

```

U-Shot 그래디언트:     linear-gradient(135deg, #3B82F6, #60A5FA)

```

그래디언트는 다음 요소에만 사용한다:
- 온보딩 화면 배경
- 홈 화면 상단 요약 카드
- 달성률 100% 축하 애니메이션
- 프리미엄 배지/라벨

일반 UI 요소(버튼, 카드, 입력 필드)에는 단색을 사용한다.

## 3. 타이포그래피

### 3.1 서체 선택

| 플랫폼 | 한글 서체 | 영문/숫자 서체 | 선정 이유 |
|--------|----------|-------------|----------|
| **iOS** | Apple SD Gothic Neo | SF Pro | 시스템 기본, 최적 렌더링 |
| **Android** | Noto Sans KR | Roboto | 시스템 기본, 가독성 우수 |
| **웹 대시보드** | Pretendard | Pretendard | 한영 혼합 시 균일한 리듬 |
| **숫자 강조** | - | Tabular Lining (등폭) | 통계 수치 정렬용 |

Flutter에서는 `fontFamily`를 지정하지 않고 시스템 기본 서체를 사용하는 것을 원칙으로 한다. 사용자가 기기 설정에서 글꼴 크기를 조정한 경우 앱이 이를 존중해야 한다.

### 3.2 타이포그래피 체계

모바일 앱 기준. `sp` 단위를 사용하여 사용자 글꼴 크기 설정을 반영한다.

| 스타일 | 크기 | 두께 | 행간 | 용도 |
|--------|------|------|------|------|
| **Display** | 32sp | Bold (700) | 40sp | 홈 화면 인사말, 핵심 수치 |
| **Heading 1** | 24sp | Bold (700) | 32sp | 화면 타이틀 |
| **Heading 2** | 20sp | SemiBold (600) | 28sp | 섹션 제목 |
| **Heading 3** | 18sp | SemiBold (600) | 24sp | 카드 제목, 서브섹션 |
| **Body Large** | 16sp | Regular (400) | 24sp | 주요 본문, 설명 텍스트 |
| **Body** | 14sp | Regular (400) | 20sp | 일반 본문 (최소 권장) |
| **Body Small** | 13sp | Regular (400) | 18sp | 보조 정보 |
| **Caption** | 12sp | Regular (400) | 16sp | 타임스탬프, 힌트 |
| **Overline** | 11sp | Medium (500) | 16sp | 라벨, 카테고리 태그 |

### 3.3 폰트 크기 원칙

- **본문 텍스트의 절대 최소 크기는 12sp**이다. 이보다 작은 텍스트는 사용하지 않는다.
- **인터랙티브 요소(버튼, 링크, 탭)**의 텍스트는 14sp 이상이어야 한다.
- **핵심 수치**(세션 수, 시간 등)는 최소 20sp 이상으로 눈에 띄게 표시한다.
- **줄 간격(Line Height)은 글자 크기의 1.4~1.6배**를 유지한다.
- **한 줄의 최대 글자 수는 40자(한글 기준)**를 넘지 않도록 한다.

### 3.4 텍스트 색상 대비

| 텍스트 유형 | 라이트 모드 (배경 white) | 다크 모드 (배경 gray-800) | 대비율 |
|------------|------------------------|--------------------------|--------|
| 프라이머리 | `#111827` | `#F3F4F6` | 15.5:1 / 13.9:1 |
| 세컨더리 | `#6B7280` | `#9CA3AF` | 5.0:1 / 5.4:1 |
| 캡션 | `#9CA3AF` | `#6B7280` | 2.7:1 (큰 텍스트만) |
| 비활성 | `#D1D5DB` | `#4B5563` | (인터랙션 불가 표시용) |

## 4. 네비게이션 구조

### 4.1 모바일 앱 탭 구성

하단 네비게이션 바를 사용하며, **최대 4개 탭**으로 구성한다. 5개를 넘기지 않는다.

```
┌──────────────────────────────────────────────┐
│                                              │
│              [화면 콘텐츠 영역]                 │
│                                              │
├──────────────────────────────────────────────┤
│  🏠 홈    📊 통계    📱 디바이스    ⚙️ 설정    │
└──────────────────────────────────────────────┘
```

| 탭 | 아이콘 | 한글명 | 영문명 | 핵심 기능 |
|----|--------|--------|--------|----------|
| 1 | 홈 | **홈** | Home | 오늘의 요약, 디바이스 상태, 빠른 접근 |
| 2 | 차트 | **통계** | Stats | 일/주/월 사용 기록, 차트, 트렌드 |
| 3 | 디바이스 | **디바이스** | Device | BLE 연결, 디바이스 정보, 동기화 |
| 4 | 설정 | **설정** | Settings | 프로필, 알림, 앱 설정, 도움말 |

### 4.2 탭 선택 이유

- **"홈" 탭을 가장 왼쪽에 배치**: 대부분의 앱에서 사용자가 기대하는 위치이다.
- **"통계"를 두 번째에 배치**: 홈에서 요약을 보고 상세를 보고 싶을 때 자연스럽게 이동한다.
- **"디바이스"를 세 번째에 배치**: 연결/동기화는 필요할 때만 접근하므로 중앙이 적절하다.
- **"설정"을 가장 오른쪽에 배치**: 관례적 위치. 자주 사용하지 않으므로 끝에 배치한다.

#### 페어링 UX 원칙

1. **자동 재연결**: 한 번 페어링한 디바이스는 앱 실행 시 자동으로 재연결을 시도한다.
2. **백그라운드 동기화**: 연결 성공 시 자동으로 미동기화 세션을 가져온다.
3. **실패 안내는 구체적으로**: "연결에 실패했습니다" 대신 "디바이스 전원을 확인하고 1m 이내에서 다시 시도해 주세요".
4. **기술 용어 회피**: "BLE", "Bluetooth Low Energy", "GATT" 같은 용어를 사용하지 않는다. "무선 연결", "디바이스 찾기" 등으로 표현한다.
5. **진행 상태 명확히**: 스캔 중, 연결 중, 동기화 중의 상태를 시각적으로 구분한다.

## 6. 인터랙션 패턴

### 6.1 버튼

#### 버튼 종류 및 크기

| 종류 | 높이 | 좌우 패딩 | 폰트 | 사용 |
|------|------|----------|------|------|
| **Primary (채움)** | 52dp | 24dp | 16sp Bold | 화면당 1개, 핵심 행동 |
| **Secondary (테두리)** | 48dp | 20dp | 14sp Medium | 보조 행동 |
| **Text (텍스트)** | 44dp | 16dp | 14sp Medium | 덜 중요한 행동 |
| **Icon (아이콘)** | 48x48dp | - | - | 네비게이션, 설정 |
| **FAB (플로팅)** | 56x56dp | - | - | 디바이스 연결 바로가기 (선택) |

#### 버튼 상태

| 상태 | 시각적 변화 | 전환 시간 |
|------|-----------|----------|
| 기본 | 지정 색상 | - |
| 눌림 (Pressed) | 밝기 -15% + 스케일 0.97 | 100ms |
| 비활성 (Disabled) | 투명도 40% | - |
| 포커스 | 외곽선 2dp | - |
| 로딩 (Processing) | 텍스트 → 스피너 | 150ms |

#### 버튼 라디우스

모든 버튼의 모서리 라디우스는 **12dp**로 통일한다. 필(pill) 형태는 세그먼트 컨트롤에만 사용한다.

### 6.2 카드 (Card)

```
패딩:       20dp (사방)
라디우스:    16dp
그림자:      라이트 모드에서만 미세한 그림자 (elevation 1)
테두리:      다크 모드에서 1dp gray-700
카드 간 간격: 12dp (세로), 12dp (가로)
```

### 6.3 입력 필드 (Input)

```
높이:       52dp
패딩:       16dp (좌우)
라디우스:    12dp
테두리:      1dp gray-300 (라이트) / gray-700 (다크)
포커스:      2dp indigo-500 / indigo-400
에러:       1dp red-500 + 하단에 에러 메시지 (12sp, red-500)
레이블:     입력 필드 위에 별도 텍스트 (14sp, gray-500)
힌트:       필드 내부 (14sp, gray-400)
```

### 6.4 로딩 상태 (Loading)

| 상황 | 패턴 | 설명 |
|------|------|------|
| 화면 첫 로딩 | 스켈레톤(Skeleton) | 콘텐츠 형태의 회색 블록이 펄스 애니메이션 |
| 데이터 새로고침 | 풀 투 리프레시(Pull-to-Refresh) | 상단에서 아래로 당기면 새로고침 |
| 버튼 동작 후 대기 | 버튼 내 스피너 | 버튼 텍스트 대신 원형 스피너 |
| 페이지네이션 | 하단 스피너 | 리스트 끝에 도달하면 자동 로드 |
| BLE 스캔 | 파동 애니메이션 | 중앙에서 퍼지는 동심원 |
| 동기화 | 프로그레스 바 | 진행률(%) + 현재/전체 건수 |

### 6.5 에러 상태 (Error)

| 에러 유형 | 표시 방식 | 메시지 예 |
|----------|----------|----------|
| 네트워크 오류 | 화면 상단 배너 (빨간) + 재시도 버튼 | "인터넷 연결을 확인해 주세요" |
| 서버 오류 | 화면 중앙 에러 일러스트 + 재시도 | "잠시 후 다시 시도해 주세요" |
| BLE 연결 실패 | 디바이스 탭 내 인라인 안내 | "디바이스 전원과 거리를 확인해 주세요" |
| 인증 만료 | 로그인 화면으로 리다이렉트 | "다시 로그인해 주세요" |
| 입력 검증 실패 | 입력 필드 하단 인라인 에러 | "이메일 형식이 올바르지 않습니다" |
| 동기화 실패 | 동기화 배너에 실패 표시 | "일부 기록을 동기화하지 못했습니다. 다시 시도" |

#### 에러 메시지 작성 규칙

1. **무슨 일이 일어났는지** 알린다 (사실)
2. **왜 일어났는지** 짧게 설명한다 (원인)
3. **어떻게 하면 되는지** 안내한다 (해결)
4. **기술 용어를 사용하지 않는다** (HTTP 코드, 에러 코드 등 표시 금지)

좋은 예: "인터넷 연결이 불안정합니다. Wi-Fi 또는 데이터를 확인한 후 다시 시도해 주세요."
나쁜 예: "Error 503: Service Unavailable"

### 6.6 빈 상태 (Empty State)

모든 목록/차트 화면은 데이터가 없을 때의 빈 상태를 명시적으로 디자인한다.

| 화면 | 빈 상태 메시지 | CTA |
|------|-------------|-----|
| 홈 (세션 없음) | "아직 사용 기록이 없습니다" | "디바이스 연결하기" |
| 통계 (기간 내 없음) | "이 기간에 사용 기록이 없습니다" | "다른 기간 선택" |
| 디바이스 (미등록) | "등록된 디바이스가 없습니다" | "디바이스 찾기" |
| 세션 목록 (비어 있음) | "표시할 세션이 없습니다" | "동기화 하기" |

빈 상태에는 반드시:
- 가벼운 일러스트레이션 또는 아이콘 (64dp)
- 1줄 메시지 (16sp)
- CTA 버튼 또는 링크

### 6.7 제스처 규칙

| 제스처 | 용도 | 적용 화면 |
|--------|------|----------|
| **탭** | 버튼, 목록 항목, 카드 선택 | 전체 |
| **좌우 스와이프** | 온보딩 페이지 전환, 차트 기간 이동 | 온보딩, 통계 |
| **풀 투 리프레시** | 데이터 새로고침 | 홈, 통계, 디바이스 |
| **스크롤** | 목록 탐색 | 전체 목록 화면 |
| **롱프레스** | 세션 삭제(Phase 2), 디바이스 선택 | 세션 목록, 디바이스 목록 |

**사용하지 않는 제스처**:
- 더블탭: 혼란을 줄 수 있으므로 사용하지 않는다
- 핀치 줌: 차트에서도 사용하지 않는다 (기간 세그먼트로 대체)
- 3D Touch / 포스 터치: 발견하기 어려우므로 사용하지 않는다

### 6.8 확인 대화상자 (Confirmation Dialog)

파괴적 행동(삭제, 로그아웃 등)에는 반드시 2단계 확인을 거친다.

```
┌──────────────────────────────────┐
│                                  │
│          로그아웃 하시겠습니까?      │
│                                  │
│  다시 로그인하면 모든 기록을         │
│  확인할 수 있습니다                 │
│                                  │
│  ┌──────────┐  ┌──────────┐    │
│  │   취소    │  │  로그아웃  │    │
│  └──────────┘  └──────────┘    │
│                                  │
└──────────────────────────────────┘
```

- 취소 버튼: 왼쪽, Secondary 스타일
- 확인 버튼: 오른쪽, Primary 또는 Danger 스타일
- 계정 삭제 시: 이메일 재입력 요구 (Phase 2)

### 6.9 토스트 메시지 (Toast / Snackbar)

성공/알림성 메시지는 화면 하단에 일시적으로 표시한다.

```

표시 시간: 3초 (에러는 5초 또는 수동 닫기)

라디우스:  12dp
배경:     gray-800 (라이트) / gray-100 (다크)
텍스트:   14sp white / gray-900
```

| 유형 | 좌측 아이콘 | 배경 | 예시 |
|------|-----------|------|------|
| 성공 | 초록 체크 | 기본 | "동기화가 완료되었습니다" |
| 정보 | 파란 i | 기본 | "새로운 세션이 3건 추가되었습니다" |
| 경고 | 노란 삼각형 | 기본 | "배터리가 20% 미만입니다" |
| 에러 | 빨간 X | 기본 | "동기화에 실패했습니다" + [다시 시도] |

## 7. 웹 대시보드 UX

### 7.1 포지셔닝

웹 대시보드는 모바일 앱의 **보조 뷰어**이다. 데이터 수집(BLE)은 불가능하며, 서버에 이미 동기화된 데이터를 넓은 화면에서 조회하는 용도이다.

| 역할 | 대상 사용자 | 핵심 기능 |
|------|-----------|----------|

### 7.2 현재 구현 상태

기존 Next.js 프론트엔드의 화면 구성 (이미 구현됨):

### 7.3 웹 대시보드 개선 방향

현재 구현은 기능적으로 완성되어 있으나, UX 가이드라인과의 정합성을 높이기 위해 다음을 고려한다.

#### 7.3.1 색상 일관성

현재 웹에서 사용 중인 Shot Type 색상:
- U-Shot: `#3B82F6` (blue-500)
- E-Shot: `#F59E0B` (amber-500)
- LED: `#10B981` (emerald-500)

이 색상은 본 가이드라인의 기능별 색상과 일치하므로 유지한다.

#### 7.3.2 Navbar 개선

현재 Navbar(`Navbar.tsx`)에는 Dashboard, Devices, Stats 3개 항목이 있다. 한글 지원 시 "대시보드", "디바이스", "통계"로 변경하되, 웹은 영문 유지도 가능하다 (사용자 언어 설정에 따름).

#### 7.3.3 모바일 반응형

웹 대시보드에서도 모바일 브라우저 접근은 가능해야 하므로, 기본적인 반응형 레이아웃을 유지한다. 현재 Tailwind의 `sm:`, `lg:` 브레이크포인트를 활용하고 있으므로 이를 유지한다.

## 9. 모션 및 애니메이션

### 9.1 기본 원칙

- 애니메이션은 **의미가 있을 때만** 사용한다 (장식용 금지).
- 사용자의 행동에 대한 **피드백**이거나, **공간적 관계를 설명**하기 위한 것이어야 한다.
- `prefers-reduced-motion` 시스템 설정을 존중하여, 접근성 모드에서는 애니메이션을 최소화하거나 제거한다.

### 9.2 전환 시간 기준

| 유형 | 시간 | 이징 | 용도 |
|------|------|------|------|
| 마이크로 인터랙션 | 100-150ms | ease-out | 버튼 눌림, 토글, 리플 |
| 화면 전환 | 250-350ms | ease-in-out | 페이지 이동, 모달 열기 |
| 콘텐츠 진입 | 300-400ms | ease-out | 카드 페이드인, 리스트 슬라이드 |
| 복잡한 전환 | 400-500ms | cubic-bezier(0.4, 0, 0.2, 1) | 온보딩 페이지, 차트 업데이트 |

### 9.3 주요 애니메이션

| 요소 | 애니메이션 | 설명 |
|------|----------|------|
| 화면 전환 | Slide Left/Right | 같은 계층 이동은 슬라이드, 하위 이동은 슬라이드업 |
| 모달 | Fade + Scale Up (0.95→1.0) | 중앙에서 약간 확대되며 등장 |
| 토스트 | Slide Up + Fade In | 하단에서 올라와 표시, 사라질 때 Fade Out |
| 스켈레톤 | Shimmer Pulse | 밝기가 좌→우로 이동하는 반짝임 |
| BLE 스캔 | Ripple (동심원) | 중앙에서 퍼지는 파동, 2초 주기 |
| 동기화 | Progress Bar | 좌→우 진행 + 완료 시 초록 체크 |
| 차트 | Draw Animation | 바/라인이 0에서 목표값까지 300ms간 성장 |
| 성취 | Confetti + Scale | 100% 완료율 달성 시 잠깐의 축하 효과 |

- 숫자 형식: 한국어 "1,234" / 영어 "1,234" (동일)
- 단위: "분", "초", "시간" / "min", "sec", "hrs"

## 부록 A: 디자인 체크리스트

새로운 화면을 설계할 때 다음 항목을 검토한다.

### 접근성 체크
- [ ] 터치 타겟이 48x48dp 이상인가?
- [ ] 텍스트 명암비가 WCAG AA를 만족하는가?
- [ ] 색상 외에 아이콘/텍스트로도 정보를 전달하는가?
- [ ] 시스템 글꼴 크기 확대 시 레이아웃이 깨지지 않는가?
- [ ] 스크린리더로 핵심 정보를 읽을 수 있는가?

### 상태 체크
- [ ] 로딩 상태가 디자인되어 있는가? (스켈레톤)
- [ ] 에러 상태가 디자인되어 있는가? (에러 메시지 + 재시도)
- [ ] 빈 상태가 디자인되어 있는가? (일러스트 + CTA)
- [ ] 네트워크 오프라인 상태를 고려했는가?

### 일관성 체크
- [ ] 버튼 스타일이 가이드라인과 일치하는가?
- [ ] 카드 라디우스/패딩이 통일되어 있는가?
- [ ] Shot Type 색상이 일관적인가?
- [ ] 폰트 크기가 타이포그래피 체계를 따르는가?

### 인터랙션 체크
- [ ] Primary 버튼이 화면에 1개만 있는가?
- [ ] 파괴적 행동에 확인 다이얼로그가 있는가?
- [ ] 풀 투 리프레시가 적용되어 있는가?
- [ ] 성공 피드백(토스트)이 있는가?

## 부록 B: 디자인 토큰 요약

Flutter 및 웹 구현 시 공통으로 사용하는 디자인 토큰이다.

### 간격 (Spacing)

```
spacing-xs:   4dp
spacing-sm:   8dp
spacing-md:   12dp
spacing-lg:   16dp
spacing-xl:   20dp
spacing-2xl:  24dp
spacing-3xl:  32dp
spacing-4xl:  48dp
```

### 라디우스 (Border Radius)

```
radius-sm:    8dp    (작은 요소: 태그, 배지)
radius-md:    12dp   (버튼, 입력 필드)
radius-lg:    16dp   (카드, 패널)
radius-xl:    24dp   (바텀시트, 모달)
radius-full:  9999dp (원형: 아바타, 아이콘 배경)
```

### 그림자 (Elevation) - 라이트 모드 전용

```
elevation-0:  없음
elevation-1:  0 1px 2px rgba(0,0,0,0.05)   (카드 기본)
elevation-2:  0 2px 4px rgba(0,0,0,0.08)   (호버/팝오버)
elevation-3:  0 4px 12px rgba(0,0,0,0.12)  (모달, 바텀시트)
```

다크 모드에서는 그림자 대신 테두리(1dp, gray-700)로 계층을 표현한다.

### 전환 (Transition)

```
duration-fast:    100ms
duration-normal:  250ms
duration-slow:    400ms
easing-default:   cubic-bezier(0.4, 0, 0.2, 1)
easing-decel:     cubic-bezier(0, 0, 0.2, 1)
easing-accel:     cubic-bezier(0.4, 0, 1, 1)
```
